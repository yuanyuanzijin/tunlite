## v0.10.1 — 2026-06-20

A small follow-up to 0.10.0, mostly about the agent skill. The bundled skill now
**proactively asks whether a tunnel should start at login/boot** before setting that
up — so a tunnel you mean to keep "always on" doesn't quietly disappear after a
reboot (neither a bare `install` nor `enable` registers autostart on its own). And
`tunlite doctor` gained a check that flags an installed skill copy that's fallen
behind the bundled one — refresh it with `tunlite install skill`.

Upgrade with `tunlite update` (or `npx tunlite@latest install`).

## v0.10.0 — 2026-06-19

This release brings tunlite closer to the `ssh` you already trust. Forwards now
speak ssh's own language — the `-L` / `-R` / `-D` flags you know — so a tunnel
reads exactly like the command you'd otherwise type by hand
(`tunlite add web --to me@host -L 8080:localhost:80`), and a single tunnel can carry
as many forwards as you need.

We've also taken the moment, while the project is still young, to settle the words
for turning tunnels on and off: `enable` / `disable` (with `restart`), each pointing
at what it acts on — a name, a `--tag`, or `all`. The names now match what they do —
a tunnel you *enable* stays up across reboots — and it's a deliberate change we'd
rather make once, now, than carry forever. A retired or mistyped verb quietly points
you to the right one.

New, too: a daemon-less `tunlite run` that keeps a single tunnel alive in the
foreground — made for containers and `systemd` units — and a quiet guard that keeps
two tunnels from claiming the same port.

Your existing tunnels keep working, untouched. Upgrade with `tunlite update`
(or `npx tunlite@latest install`).
